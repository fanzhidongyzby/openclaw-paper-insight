# 论文公众号解读技能

将学术论文转化为**微信专业公众号风格**的深度解读文章。

## 设计理念

融合学术解读和公众号传播两个流派：

1. **原子管线设计**：
   - 原子管线：split → squeeze → plain → feynman → napkin sketch → advisor review
   - 零术语规则：每个技术概念必须立刻落在读者见过的事情上
   - 承重概念场景化：先生活场景（3级渐进），最后才贴技术名字
   - 认知碰撞卡片：ASCII art 直观展示论文洞见与个人认知的碰撞
   - 餐巾纸速写：一张图看清新旧框架的结构位移

2. **公众号风格**：
   - 📱 标题层级清晰，适合手机阅读
   - 💬 流畅自然，对话式表达，像和朋友聊天
   - 🎨 视觉丰富，emoji、加粗、分隔线
   - 📖 故事性，有吸引力的前言
   - 📊 数据支撑，完整表格、实验数据
   - 🎯 深入浅出，生活化类比
   - 💡 互动感，用问句让读者有参与感

3. **零割裂感**：
   - 输出是一篇文章，不是几份报告拼接
   - 原子在后台运行，读者看到的是一段连贯的认知旅程

## 核心原则

### 1. 零术语规则

**承重概念**：论文去掉就塌了的核心概念，不能括号注释一笔带过。每个承重概念必须场景化：

**示例（坏）**：
> 共现概率只取决于隐含空间中的距离（平移对称性）

**示例（好）**：
> 一月和二月经常出现在同一段话里，一月和七月就很少——距离越近越常一起出现。这就是"平移对称性"：位置平移不改变相对关系。

**场景化三步法**：
1. 先找一个读者亲身经历过的场景（群聊记录、排队、找路...）
2. 在场景中逐步展开——至少 3 级渐进，让读者在熟悉的事情上"感受到"概念在变化
3. 最后才贴技术名字。此时名字是标签，不是定义

**判断标准**：去掉技术名字，读者仍然知道你在说什么 = 落地成功。只剩括号注释 = 失败。

**普通术语**：一句话类比即可，不需要场景展开。区分承重与非承重是关键判断。

### 2. 核喻（Plain Method）

论文最难 grok 的部分是方法论。ASCII 图给了结构轮廓，但结构不等于可理解。这一步用核喻方法，为核心机制找到一个结构同构的日常类比。

- **抓动词结构**：这个方法在做什么操作？什么作用于什么？什么流向什么？
- **找核喻**：在日常经验中找一个有相同动词结构的东西。核喻必须承重——方法的每个关键组件都能映射到类比的一个部分。
- **喻到底**：沿着核喻把方法从头到尾走一遍。读者跟着类比走完，应该能自己复述方法的逻辑。

核喻嵌在「核心机制」段落中，紧跟 ASCII 图之后。不是单独一节，是同一段话的自然延续：「用图画完了，现在用人话说一遍。」

### 3. 认知碰撞卡片

**卡片 = ASCII art，视觉优先**。

好卡片的标准：遮住文字只看线条，仍能感受到结构关系（分叉、汇聚、层级、对比、拉扯）。

反面教材——文字列表伪装成卡片：

```
+------------------+
| 论文说: X        |
| 我原来想: Y      |
| 现在变成: Z      |
+------------------+
```

这不是卡片，是带框的文字。结构关系要用空间布局表达，不是用标签声明。

**每张卡片锚定一个具体场景**：一个思考方式、一个决策场景、或一个认知盲区的改变。卡片下方附一句金句级启发——能脱离上下文单独成立。

**碰撞形状**（选跟碰撞匹配的空间结构）：

| 形状 | 示例 | 适合场景 |
|------|------|---------|
| 分叉型 | YES/NO 路径分叉 | 两种选择，结果不同 |
| 张力型 | 拉扯两端 | 观点冲突，需要平衡 |
| 阈值型 | 非线性转折点 | 过了某个值才有效果 |
| 缺口型 | 中间空白 | 现有方案缺失的部分 |
| 翻转型 | 上下颠倒 | 常规认知被颠覆 |

**好卡片示例（分叉型）**：
```
  Soul: "做了才懂"
        |
        |  但如果结构藏得太深...
        v
  +------------------+
  | 反馈能暴露结构?   |
  +---+----------+---+
      |          |
     YES         NO
      |          |
   生成优先    降秩优先
  (原路径)   (先看再做)
      |          |
      |     CTA: 0% vs 94%
      v          v
   Soul 成立   Soul 需加边界
```

**阈值型示例**：
```
  我以为: 投入越多，产出越多（线性）

  投入 ----+----+----+----+----+---->
           |    |    |    |    |
  产出     .    .    .    .    .   线性预期
           .    .    .   /
  实际     .    .    . _/         <-- 阈值
           .    .    ./
           .____.___.'............   无效区
           |
       "原来这段全是白费的"
           |
           v
  决策改变: 不追问"做了多少"
            要追问"过没过线"
```

### 4. 餐巾纸速写（Napkin Sketch）

增量看清了，现在需要一张图。想象你在咖啡馆，拿起餐巾纸给朋友画：「以前大家是这么想的，这篇论文说应该这么想。」

这张图的目标：**让读者一眼看出思想框架的位移**——不是细节对比，是 gestalt 级别的「啊，原来整个思路变了」。

怎么画：
- **识别主流框架**：这篇论文之前，这个方向上大家默认的思考方式/架构/范式是什么？可能是一个、也可能是两三个并存的主流方案。
- **识别本文框架**：这篇论文提出的新框架/新视角/新架构是什么？
- **画对比**：用 ASCII 图把两者并排或上下放，标注关键差异。图要能自解释——不看正文也能大致理解位移方向。
- **标注位移**：在图上或图下用一句话点出：从 X 到 Y，核心位移是什么。

画图原则：
- 优先用结构对比（左右并列 / 上下对照），不是功能清单
- 框架之间的「结构差异」比「功能差异」重要——不是列 feature 对比表，是画出思考方式的不同形状
- 越简洁越好。餐巾纸不大，能画三五个框就够了

### 5. 博导审稿（白话审）

换个身份：你是这个方向上带了二十年研究生的博导。一个学生拿着这篇论文来找你，说「老师您看看这个」。你不是在填审稿表格，你是在判断——这篇东西值不值得认真对待。

用白话说，不用学术腔。像是在办公室里跟学生聊：

- **选题眼光**：这个问题值得做吗？是真缺口还是人造缺口？这个方向上的人都在往哪儿走，这篇论文是跟着走还是在开路？
- **方法成熟度**：方法是巧劲还是蛮力？有没有比这更自然的做法被忽略了？技术路线图画得清不清楚？
- **实验诚意**：实验是在证明方法好，还是在证明作者聪明？baseline 选得公不公道？消融做到位了没有？数字经不经得起追问？
- **写作功力**：论文有没有在最该说清楚的地方偷懒？哪一段如果重写会让整篇论文升一个档次？
- **一句话判决**：如果这篇论文投到你手上审，你会给什么意见？strong accept / weak accept / borderline / weak reject / strong reject？一句话说理由。

不为了挑刺而挑刺。博导见过太多论文，知道什么是真问题什么是鸡蛋里挑骨头。好的地方要说好，差的地方要说出差在哪儿、怎么能更好。

### 6. 公众号风格

**结构**（手机友好）：
- 前言（吸引眼球，像朋友聊天）
- 研究缺口（论文要填哪条缝）
- 核心增量（一句话说清 before vs after）
- 方法论（核心机制图 + 核喻 + 代码示例）
- 关键概念（费曼讲解）
- 餐巾纸速写（新旧框架对比）
- 实验结果（完整表格 + 对比 + 分析）
- 博导审稿（内行判决）
- 认知碰撞（如有）
- 总结与启示（给读者的价值）

**表达风格**：
- 🎨 **视觉丰富**：emoji、加粗、分隔线
- 💬 **流畅自然**：对话式，像和朋友聊天
- 📖 **故事性**："想象一下这些场景"、"你可能会问"
- 🎯 **深入浅出**：生活化类比（团队协作、电影推荐、学术写作）
- 💡 **互动感**："这说明了什么"、"你可能会问"、"让我们想想"

**格式要求**：
- 标题用 `#`（一级标题）
- 章节用 `##`（二级标题）
- 小节用 `###`（三级标题）
- 关键信息用 `**粗体**`
- 重要引用用 `>`
- 分隔用 `---`
- ASCII 图使用纯字符，禁止 Unicode 绘图符号

### 7. 写作卫生检查

综合输出完成后，对全文执行轻量扫描，只查最高频的 AI 写作惯性：

1. **否定式排比**："不是...而是..."、"不是...是..."、"不再是...而是..."——全文上限两处。超标的改为直接陈述正面内容。
2. **三段式列举**：连续三项并列的改为两项或四项。
3. **破折号过度**：同一段落超过两个破折号的，换成逗号或句号。

扫描完后列出修改清单（哪句触发哪条，改前→改后）。清单确认后最终输出。

## 执行步骤

### 步骤 1：接收论文并预处理

| 用户输入 | 转换规则 |
|---------|---------|
| `2601.01290` 或 `arxiv:2601.01290` | → `https://arxiv.org/html/2601.01290` |
| `https://arxiv.org/abs/...` 或 `.../pdf/...` | → 替换为 `/html/` |
| `https://arxiv.org/html/...` | 直接使用 |
| 其他 URL 或 PDF 路径 | 按原有逻辑处理 |

**arxiv 论文一律转为 `/html/` 格式**，HTML 版本可直接抓取全文。

提取标题、作者、发布日期，填入文章头部。

### 步骤 1.5：动态分片（新增）

**关键原则**：避免固定分段导致上下文撑爆，必须根据当前上下文使用率动态计算分段大小。

#### 1.5.1 检测当前上下文状态

```bash
# 检查当前上下文使用情况
session_status
```

获取以下信息：
- 总上下文容量（例如：166k tokens）
- 当前已用量（例如：1.6k）
- 已用百分比（例如：1%）

#### 1.5.2 动态计算分段参数

```
安全剩余空间 = 总容量 - 当前已用 - 预留空间（50k tokens）
每段目标字符数 = 安全剩余空间 × 0.4（预留空间给推理和生成）
```

**中文文本估算**：1 字符 ≈ 1 token

#### 1.5.3 分段策略表

| 当前上下文使用率 | 每段字符数 | 分段数建议（10万字论文） |
|-----------------|-----------|------------------------|
| < 30%           | 20,000    | 5-6 段                 |
| 30-50%          | 12,000    | 8-10 段                |
| 50-70%          | 8,000     | 12-15 段               |
| 70-85%          | 5,000     | 20-25 段               |
| > 85%           | 3,000     | 30+ 段                 |

#### 1.5.4 分段执行流程

1. **获取全文长度**：先快速扫描文档头/尾，估算总字符数
2. **计算分段数**：分段数 = ceil(总字符数 / 每段目标字符数)
3. **逐段抓取**：每段抓取后，重新执行 `session_status` 检查上下文
4. **动态调整**：如果上下文使用率超过 70%，自动缩小下一段大小（×0.7）

**示例**：
```
论文：https://arxiv.org/html/2601.01290
1. 检测上下文：已用 1.6k/166k (1%)
2. 计算：剩余 ≈ 164k，预留 50k → 每段 45k 字符
3. 但由于使用率 < 30%，按策略表取 20,000 字符/段
4. 分段抓取：每段 20k 字符，共约 8-10 段
```

**警告红线**：如果单次处理后上下文使用率超过 90%，必须：
1. 立即停止处理
2. 通知用户：上下文已满，建议新建会话或继续缩小分段

### 步骤 2：原子管线（内部执行，不分别输出）

**重要**：分段后，在执行原子管线前，必须重新检查上下文状态，确保不会撑爆。

#### 2a. Split（拆）——研究缺口

论文有固定骨架，但骨架下面藏着真正的结构：

- **缺口**：已有研究走到了哪里？哪里还没做到？这篇论文声称填的是哪条缝？
- **假设**：作者做了哪些显式和隐式的假设？
- **方法**：用什么方法填这条缝？方法的核心机制是什么？（不是复述 method section，是抓住方法的动词结构——它在做什么操作？）
- **证据**：用什么数据/实验来证明方法站得住？主要结果是什么？
- **贡献声明**：作者自己说他贡献了什么？

画出论文的逻辑拓扑：从缺口到方法到证据到结论的路径。

#### 2b. Squeeze（榨增量）——核心增量

论文最值钱的部分是增量——before this paper vs after this paper，世界多了什么？

- **增量是什么？** 一句话说清楚：这篇论文让我们知道了什么以前不知道的？
- **核心机制图**：用 ASCII 图画出方法的内部结构——组件是什么、数据怎么流、操作之间什么关系。这张图是方法的 X 光片，看完知道「它内部怎么转的」。（注意：与 Napkin Sketch 不同，这里画的是方法自身的内部结构，不是新旧框架对比。）

#### 2c. Plain（白话方法）——核喻

ASCII 图给了结构轮廓，但结构不等于可理解。这一步用核喻方法，为核心机制找到一个结构同构的日常类比。

- **抓动词结构**：这个方法在做什么操作？什么作用于什么？什么流向什么？
- **找核喻**：在日常经验中找一个有相同动词结构的东西。核喻必须承重——方法的每个关键组件都能映射到类比的一个部分。
- **喻到底**：沿着核喻把方法从头到尾走一遍。读者跟着类比走完，应该能自己复述方法的逻辑。

核喻嵌在「核心机制」段落中，紧跟 ASCII 图之后。不是单独一节，是同一段话的自然延续：「用图画完了，现在用人话说一遍。」

#### 2d. Feynman（关键概念）

论文中往往有 1-3 个关键概念，是理解整篇论文的前提。可能是论文自己提出的新术语，也可能是领域内已有术语但对非专业读者不透明。

用费曼技巧讲清楚：

- **识别**：哪些概念是理解这篇论文的钥匙？不超过三个。
- **从零建起**：假设读者完全不懂这个概念，从最基础的地方开始讲。不用术语解释术语。
- **用例子落地**：每个概念给一个具体例子，让抽象变成可感知的。
- **检验**：讲完之后，读者能不能用自己的话复述这个概念？如果不能，说明没讲透，重来。

#### 2e. Napkin Sketch（餐巾纸速写）

增量看清了，现在需要一张图。想象你在咖啡馆，拿起餐巾纸给朋友画：「以前大家是这么想的，这篇论文说应该这么想。」

这张图的目标：**让读者一眼看出思想框架的位移**——不是细节对比，是 gestalt 级别的「啊，原来整个思路变了」。

怎么画：

- **识别本文框架**：这篇论文提出的新框架/新视角/新架构是什么？
- **画对比**：用 ASCII 图把两者并排或上下放，标注关键差异。图要能自解释——不看正文也能大致理解位移方向。
- **标注位移**：在图上或图下用一句话点出：从 X 到 Y，核心位移是什么。

#### 2f. 博导审稿（白话审）

换个身份：你是这个方向上带了二十年研究生的博导。一个学生拿着这篇论文来找你，说「老师您看看这个」。

用白话说，不用学术腔：
- **选题眼光**：这个问题值得做吗？是真缺口还是人造缺口？

- **一句话判决**：strong accept / weak accept / borderline / weak reject / strong reject？一句话说理由。

### 步骤 3：编织输出

将原子的中间结果编织成一篇连贯的分析。按以下结构输出：

**编织原则**：
- 「缺口」段自然过渡到「增量」：知道了缺口在哪，自然想看这一步到底迈了多远。
- 「增量」中核心机制自然过渡到「关键概念」：机制看清了，想把几个关键术语彻底搞懂。
- 「关键概念」自然过渡到「Napkin Sketch」：概念都懂了，自然想用一张图看清新旧框架的结构差异。
- 「Napkin Sketch」自然过渡到「博导审稿」：图看完了，位移一目了然，自然想听一个内行怎么评。
- 「博导审稿」自然过渡到「启发」：博导评完了，知道哪里实哪里虚，自然想问——这篇论文的思想撞进我的体系，哪里会变形？

### 步骤 4：写作卫生检查

综合输出完成后，对全文执行轻量扫描，只查最高频的 AI 写作惯性：

1. **否定式排比**："不是...而是..."、"不是...是..."、"不再是...而是..."——全文上限两处。超标的改为直接陈述正面内容。
2. **三段式列举**：连续三项并列的改为两项或四项。
3. **破折号过度**：同一段落超过两个破折号的，换成逗号或句号。

扫描完后列出修改清单（哪句触发哪条，改前→改后）。清单确认后进入步骤 5。

### 步骤 5：生成文件

1. 生成 Markdown 格式的公众号文章
2. 文件命名：`{论文标题}-解读.md`
3. 保存到工作区：`/data/openclaw/workspace/`

## 输出质量标准

- **缺口要准**：不是复述 abstract，是用自己的话说清楚研究边界在哪、缺口是什么。
- **增量要锐**：一句话说出 before vs after。如果说不出，说明没找到真增量。
- **机制要可视**：用 ASCII 图画出方法的内部结构（组件、数据流、操作关系）。这是方法的 X 光片——看完知道「它内部怎么转的」。与 Napkin Sketch（新旧框架对比地图）侧重点不同。
- **核喻要承重**：方法的核喻不是装饰——去掉它，读者就回到了看图发呆的状态。好的核喻让读者跟着走完就能复述方法的逻辑。
- **概念要费曼**：讲完关键概念，读者能用自己的话复述。如果还需要回头查术语表，说明没讲透。
- **速写要一眼**：Napkin Sketch 的标准是「不看正文也能大致理解位移方向」。如果图需要配大段文字才能看懂，说明图没画好。结构对比优于功能清单。
- **博导要像博导**：不是清单式挑刺，是一个见过太多论文的内行在白话点评。有判断力，有分寸感，该夸的夸该批的批，最后给一句判决。
- **启发要私人**：不是泛泛的"可迁移思路"，是具体指出读者体系里哪个零件可以升级、哪个盲区被照亮。没有实质冲击就诚实说没有——硬挤的启发比没有更差。
- **零割裂感**：读完整篇分析，像一个人在跟你讲「我读了篇论文，让我告诉你它干了啥、好不好、对我有什么用」。
- **公众号风格**：emoji、对话式、视觉丰富、手机友好。

## 约束

### L0: 通用约束

1. **语言**：永远使用中文表达
2. **代码块**：代码示例使用 ``` 代码块
3. **表格**：使用 Markdown 表格格式
4. **Emoji 使用**：适量使用，增强视觉效果，不滥用
5. **文件命名**：使用中文文件名
6. **ASCII Art 约束**：使用纯 ASCII 字符绘制，禁止 Unicode 绘图符号（如 ─ │ ┌ ┐ └ ┘ 等）

### L1: 内容约束

1. **零术语原则**：每个技术概念必须有生活化类比
2. **场景化三步法**：承重概念必须用 3 级渐进的场景化解释
3. **诚实原则**：没有碰撞就不造卡片，不编造启发
4. **数据完整**：实验数据必须完整，不能只给结论
5. **中文优先**：标题、章节、示例都用中文
6. **写作卫生**：完成输出后必须执行写作卫生检查
